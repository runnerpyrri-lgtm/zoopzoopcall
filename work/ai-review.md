# 라운드 2 — 목록 masthead 접근성 (agent/mobile-list-masthead)

## 점검 기준

- 기준 저장소: `robom-labs/homebom`. 기준 브랜치: `agent/mobile-alert-access`(Draft PR #14) 위에 쌓은 `agent/mobile-list-masthead`.
- PR #14가 상세 화면 알림 접근성(순서 재배치·권한 정합성·44px 스위치·목록 선행 권한배너 숨김)을 이미 해결한 상태에서, 목록(첫 진입) 화면의 남은 세로 점유 문제를 다룬다.
- 모바일 실측(#14 적용 기준, VITE 개발 서버 + Chromium 실측):
  - 360×800 / 320×568 목록에서 masthead 높이 124px, 첫 공고 카드 top 322px.
  - 상세 알림 카드 top 422px, 스위치 58×44(#14 유지). 가로 넘침 없음.

## 리드(Claude Code) 발견 및 제안

- 문제와 근거: `ListScreen.tsx`의 masthead(브랜드 "청약봄" 32px + source 배지 + 태그라인 문단)가 세로 124px를 차지해, #14 적용 후에도 핵심 행동(공고 카드 탭)이 top 322px에 위치. 320×568에서는 카드 앞 스택이 화면의 57%.
- 사용자 영향도: 중간. 매일 여닫는 알림앱 특성상 첫 진입 시 카드 도달이 늦어지는 누적 불편.
- 모바일 터치·스크롤 편의성 영향: 첫 카드를 위로 올려 첫 진입 순간의 스크롤 부담을 낮춤. 터치 타깃은 이미 44px라 이 변경으로 바뀌지 않음.
- 구현 난이도·회귀 위험: 낮음. `styles.css` 로컬 CSS 값 조정뿐. `packages/core`·라우팅·`/homebom/` base path 무관.
- 초기 제안: `.masthead` padding 18px→10px, `.masthead__tagline` 15px→13.5px.

## 담당자 독립 분석 및 교차검토

# 라운드 4 — 카드 밀도·문구·주택 형태·총/모집 세대 구분 (agent/compact-notice-cards)

- 리드 보완: APT 전용 API는 `아파트`로 정규화하고, 전체 세대수는 값이 있을 때만 숫자를 표시한다. 결측이면 `총 세대수 공고문 확인`으로 알려 숫자를 꾸며내지 않는다.

# 라운드 4 — 필터·상태 바 모바일 최적화 (agent/filter-status-bar-optimize)

## 사용자 요청

- 목록 상단 필터·상태 바가 답답하고 최적화가 별로. 폰에서 한 번에 깔끔하게 다 보이게(a) 하거나 버튼식으로 펼치게(b). 간격이 너무 붙어있음. "생각해서 최적화."

## 점검 기준 / 현황(실측)

- 360·320px에서 타입 칩 4개(전체/무순위/잔여세대/취소후재공급) 중 뒤 2개가 가로 넘침으로 화면 밖(스크롤 어포던스 없음), 지역 select가 `max-width:36vw`로 "전체 지"/"전체 :"로 잘림, 상태 세그먼트도 넘쳐 "마감·취소" 잘림. 두 줄 간격이 좁음.

## 담당자 독립 분석 및 교차검토

### Planner
- 옵션 (a) 채택, CSS 전용(FilterBar props·로직·마크업 무수정). 합격 기준 제시(320/360에서 칩4·상태3·지역 라벨 전부 잘림 없이).
- 반박·보완: 칩 wrap 2줄로 세로 증가 → 지역 select를 독립 행으로 내려 칩이 폭을 온전히 쓰게. 상태를 primary로 위로 올리는 재배치는 마크업 변경이라 범위 밖(별도 MINOR).

### Growth
- (a) 채택. 우선순위 상태 > 유형 > 지역. 긴급성 앱은 필터 발견성이 전환의 전부 → 접기(b)는 마찰.
- 반박·보완: 상태를 primary 세그먼트로 상단 크게 두는 편이 이상적(단 재배치는 구조 변경). "잘림=스크롤 힌트" 주장은 세로 목록에서 가로 스크롤 발견성 낮아 기각.

### Architect
- FilterBar props/tablist·tab·aria-selected/counts 바인딩/STATUS_ORDER 자동전환 불변. 회귀점: `.filters` 음수 margin bleed, segmented grid 전환 시 `overflow-x`/`flex:none` 반드시 제거(안 하면 재넘침), 마감·취소 nowrap.
- 반박·보완: **상태를 top primary로 올리지 말 것**(statusView가 목록 heading·자동전환 주축이라 유형→상태 위계 유지가 정보구조상 맞음). region 칩화 제외(지역 수 가변). sticky 보류.

### Inspector
- 기준선 typecheck/test72/build PASS, base path·secret·DB·lockfile 무관.
- 조건: 320px 칩-select 겹침, 세그먼트 grid 전환 시 aria 보존, 다크모드·저높이(320×568) 캘린더 밀림 수동 검증. 시각 회귀 테스트 부재.

## 최종 리드 판단

### 채택 (CSS 전용, 순서 유지 = type → region → status)
1. 타입 칩: `.filters__chips` `flex-wrap:wrap` + `flex:1 1 100%`, `overflow-x` 제거 → 4칩 전부 표시(좁으면 2줄).
2. 지역 select: 칩이 전폭을 쓰면서 select가 자연히 아래 독립 행으로, `max-width:36vw` → `none`으로 "전체 지역" 전체 라벨 표시.
3. 상태 세그먼트: `.segmented` `display:grid; grid-template-columns:repeat(3,1fr)`, `overflow-x`·`flex:none` 제거, 내부 `justify-content:center`·`white-space:nowrap` → 3개 한 줄에 잘림 없이.
4. `.filters`를 세로 flex + `gap:13px`, top-row `row-gap:10px`로 간격 확대(답답함 해소). 터치 48px 유지, aria/마크업 불변.

### 보류/제외 (팀 합의)
- 옵션 (b) 접기 버튼식: 고빈도 필터에 탭 마찰 → 제외.
- 상태 세그먼트를 최상단 primary로 재배치(growth 제안): 마크업/정보구조 변경 → 별도 MINOR로 분리(architect·planner 반대).
- region 칩화, sticky 필터: 제외/보류.

### 합격 기준 및 실측 결과
- 320·360px: 타입 칩 4개 전부 노출, 지역 "전체 지역" 전체 표시, 상태 "접수 중 1 / 접수 예정 2 / 마감·취소 0" 한 줄 잘림 없음, 가로 스크롤/넘침 없음(overflowX=false), 그룹 간 간격 확대.
- 다크모드 정상, 터치 48px 유지, aria(tablist/tab/aria-selected) 마크업 불변.
- typecheck PASS, test 72 PASS(core 55 + web 17), build PASS, dist `/homebom/` base 유지. diff는 styles.css 1파일(CSS 전용).

# 라운드 5 — 모바일 도달성·접수 시작 09:00 알림·검증된 단지 규모 (agent/mobile-schedule-complex-data)

## 발견한 문제와 근거

- 첨부 화면과 실제 320×568 점검에서 앱바 84px, 필터 223px, 월 캘린더 349px 뒤에 첫 공고가 약 769px부터 시작했다. 화면 첫 진입에 핵심 공고와 행동이 보이지 않아 한 손 탐색 부담이 컸다.
- 앱바의 `src="/bom-homebom.svg"`는 GitHub Pages의 `/homebom/` base 경로에서 이미지가 깨져, 첨부 화면의 `청약□봄`처럼 보였다. 점검 중 main의 별도 PR이 인라인 마크로 이미 고쳤고, 리베이스 때 그 수정을 보존했다.
- 현재 알림은 15초 간격 확인과 화면 복귀만 사용했다. 접수 시작 시각은 정규화 과정에서 KST 09:00으로 이미 고정되지만, 가장 가까운 예약 시각에 맞춰 깨우는 타이머가 없었다.
- 원본 청약 API의 공급 세대는 이번 모집 규모다. 루원시티처럼 단지 전체 1,789세대 중 9세대가 나오는 맥락을 알려주려면 별도의 공개 단지 자료를 주소와 함께 대조해야 했다.

## 담당자 독립 분석과 교차 검토

### 모바일 UX 담당

- 월 캘린더와 유형·지역 필터를 모바일에서 요약 버튼으로 접고, 접수 상태 세그먼트는 첫 화면에 남기는 안을 제안했다. 44px 이상 터치 영역을 유지하고 데스크톱에서는 모두 기본 펼침으로 둔다.
- Claude의 최신 `filterbar-optimize` 기록은 고빈도 필터를 모두 바로 노출하자는 의견이었다. 잘림을 없앤 점은 채택했지만, 320px에서 카드 도달성이 더 나빠지는 것을 실측해 모바일만 접는 방식으로 보완했다.

### 알림 안정성 담당

- 러닝봄의 긴 대기 타이머 재무장, 화면 복귀 재확인 패턴을 확인했다. 청약봄은 날짜만 있는 알림이 아니라 접수 시작을 KST 09:00으로 정규화하므로, 여러 시간대를 추측하지 않고 정확한 시작 시각 타이머를 추가하는 편이 안전하다.
- 15초 점검과 visibility 재확인은 안전망으로 유지한다. 운영체제가 백그라운드 타이머를 보장하지 않으므로 앱이 완전히 종료된 상태의 푸시 보장은 주장하지 않는다.

### 데이터 정확성 담당

- 이름만으로 단지 규모를 붙이면 동명 단지 오매칭 위험이 있다. 단지명 정규화 뒤 주소 토큰까지 함께 대조하고, 확인 가능한 8개 공고만 보강하며 그 외에는 기존 `공고문 확인` 안내를 유지하자고 제안했다.
- 루원시티2차 SK Leaders' VIEW의 아파트 1,789세대와 오피스텔 529실을 합산한 2,318을 아파트 총세대로 표시하지 않는다는 반대 의견을 채택했다.

### 품질 담당

- 워드마크 base 경로, 화면 접힘의 키보드·ARIA 상태, 타이머의 가장 이른 알림 선택, 단지 규모 보강의 기존 서버 값 보존을 회귀 지점으로 지정했다.
- 처음 타이머 테스트는 비정렬 배열에서 늦은 알림을 고르는 결함을 발견했고, 최솟값 선택으로 수정한 뒤 테스트를 통과시켰다.

## 구현과 검증

1. 모바일은 상태 세그먼트 → `유형·지역 필터` 토글 → 월 일정 요약 순서로 압축했고, 데스크톱은 기존처럼 펼쳐 둔다. 워드마크는 main의 인라인 마크 수정을 유지해 깨지지 않는다.
2. 상세 알림에 `접수 시작일 7월 N일 09:00 기준` 문구와 시작 시각을 표시한다. 가장 가까운 예약 시각에 맞춘 타이머와 긴 대기의 재무장을 추가했다.
3. 단지 규모 프로필은 core와 Edge Function에 같은 주소 대조 규칙으로 넣었다. 기존 서버가 준 총세대수는 덮어쓰지 않는다.
4. core 59건과 web 19건, 합계 78건 테스트, 타입체크와 production build를 통과했다. 로컬 dev 서버에서 `/homebom/bom-homebom.svg`가 200으로 응답하고 새 모바일 접힘 소스가 제공되는 것을 확인했다.
5. 이 실행 환경에서는 브라우저 자동화 도구가 이어지지 않아 마지막 토글 변경 후의 새 스크린샷은 생성하지 못했다. 변경 직전 320px 실측에서 캘린더를 요약한 뒤 첫 카드가 약 454px로 올라온 것은 확인했으며, 이번 필터 접힘은 그 위 공간을 더 줄이는 CSS·컴포넌트 변경이다. PR CI와 실제 모바일 브라우저에서 마지막 확인이 필요하다.

## Claude Code 협업 상태

- 비공개 회의록의 최신 Claude 필터 의견을 읽고 채택·반박 근거를 위에 반영했다. 새 라운드 검토 요청도 같은 회의록 저장소에 남겼다.

# 라운드 6 — 유형 필터 2열/4열 반응형 정렬 (agent/responsive-type-filter-grid)

## 발견한 문제와 근거

- 첨부한 모바일 화면에서 `전체 · 무순위 · 잔여세대` 뒤에 `취소후재공급`만 단독 행으로 떨어지고, 지역 선택과 상태 필터까지 이어져 첫 공고가 늦게 보였다.
- 현재 데이터 모델의 실제 공고 유형은 `무순위`, `잔여세대`, `취소후재공급` 세 가지다. `불법행위`는 현재 청약홈 정규화 모델에 없으므로 빈 필터를 추가하면 오히려 오해를 만든다.

## 담당 관점과 최종 판단

- 모바일 UX 관점은 2×2 동일 폭 버튼이 단독 행보다 시선 이동과 엄지 이동을 줄인다고 봤다. 터치 높이 48px을 유지한다.
- 정보 구조 관점은 유형을 숨기는 토글보다 사용자가 요청한 대로 항상 보이는 편이 낫다고 판단했다. 월 일정 요약은 그대로 두어 세로 부담을 상쇄한다.
- Claude의 최신 기록은 고빈도 필터 상시 노출을 권했다. 이번에는 그 의견을 채택하되, 기존 flex wrap 대신 명시적 grid로 단독 행 문제를 해결했다. 새 Claude 응답은 세션 한도로 받지 못했다.
- 리드 판단은 700px 미만 2열, 700px 이상 4열+지역 선택 한 줄이다. 600px대 태블릿에서 긴 `취소후재공급`이 답답해지는 것을 피했다. 향후 API가 추가 유형을 내놓으면 같은 grid가 자동으로 다음 행을 만든다. 현재 존재하지 않는 유형은 제외한다.

## 구현과 위험 관리

1. `FilterBar`에서 모바일 전용 접기 상태를 제거해 유형·지역을 항상 같은 정보 계층으로 보여준다. tablist, tab, `aria-selected`, 기존 클릭 콜백은 유지한다.
2. 유형은 700px 미만에서 2열, 700px 이상에서 4열 동일 폭 grid다. 지역 select는 좁은 화면에서는 전폭, 넓은 화면에서는 유형과 같은 줄의 고정 폭이다.
3. 글자 크기는 모바일 15px, 넓은 화면 14px으로 조절했고 모든 버튼은 48px 이상이다. 가로 overflow 방지를 위해 grid item의 `min-width: 0`과 select 전폭을 적용했다.
4. 타입체크, 78개 테스트, production build를 통과했다. 이 실행 환경에는 agent-browser 명령이 없어 마지막 실제 브라우저 스크린샷은 수행하지 못했다.
