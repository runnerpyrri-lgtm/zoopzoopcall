# 청약봄 전체 점검 AI 협업 기록

## 점검 기준

- 기준 저장소: `robom-labs/homebom`.
- 기준 브랜치: `main`에서 새로 만든 `agent/mobile-alert-access`.
- 관련 변경: PR #13의 상태 세그먼트 최소 높이 44px 변경은 2026-07-11 병합 완료.
- 모바일 실측:
  - 360×800 목록에서 첫 공고 카드 `top 387 / bottom 650`, 하단 내비 `top 735`.
  - 320×568 목록에서 첫 공고 카드 `top 435 / bottom 753`, 하단 내비 `top 502`.
  - 360×800 상세에서 외부 링크 `top 342~509`, 알림 카드 `top 620`, 스위치 `58×34`, 권한 버튼 `bottom 741`, 하단 내비 `top 735`.
  - 320×568 상세에서 알림 카드 `top 620`, 하단 내비 `top 502`.
- 성능 실측: TTFB 19.4ms, FCP 140ms, LCP 1.408s, CLS 0.

## 독립 분석

### Architect

- 문제와 근거: 상세 알림 카드가 외부 링크·안내문 뒤에 있어 고정 내비와 겹치며, `.switch`는 34px이고 접근 가능한 이름이 없다. `DetailScreen.tsx`, `styles.css` 근거.
- 사용자·터치 영향: 핵심 알림 행동이 늦게 보이고 44px 기준에 미달한다.
- 난이도·위험: 상세 순서와 히트박스 변경 S~M, 전역 내비 재설계는 위험이 커 제외.
- 초기 의견: 알림 카드 상향, 권한 성공 후 구독, 스위치 44px·이름 채택. 목록 배너 제거·CSS 분할 보류.

### Builder

- 문제와 근거: 권한 요청 결과를 확인하지 않고 `subscribe()`를 실행하며 `PermissionBanner`가 독립 상태를 가진다.
- 사용자·터치 영향: 스위치는 켜졌지만 실제 알림은 오지 않는 거짓 성공 상태가 생긴다.
- 난이도·위험: 권한 흐름 M, 스위치 CSS S.
- 초기 의견: `DetailScreen.tsx`, `PermissionBanner.tsx`, `styles.css` 안에서 수술식 수정. 외부 링크와 안내문 순서만 최소 조정.

### CEO Orchestrator

- 문제와 근거: 앱 핵심 가치인 알림 설정이 상세 첫 화면 하단으로 밀려 있고 목록은 맥락 없이 권한을 먼저 요청한다.
- 사용자·터치 영향: 전환과 신뢰를 동시에 저해한다.
- 난이도·위험: 권한·순서 변경 S~M, 고정 CTA는 기존 내비와 충돌.
- 초기 의견: 권한-구독 정합성, 알림 카드 상향, 스위치 접근성을 한 목표로 처리.

### Growth Marketer

- 문제와 근거: 목록에서 공고를 보기 전에 권한을 요구하고, 상세에서 외부 사이트 링크가 알림 행동보다 먼저 나온다.
- 사용자·터치 영향: 권한 거절과 외부 이탈 가능성이 높다.
- 난이도·위험: 목록 배너 제거 S, 상세 흐름 변경 M.
- 초기 의견: 목록 선행 권한 배너 제거와 상세 알림 CTA 상향 채택.

### Inspector

- 문제와 근거: 현재 CI는 타입체크·단위 테스트·빌드만 수행해 실제 360×800 겹침을 잡지 못한다. PWA base path 검증도 `index.html` 문자열 한 항목뿐이다.
- 사용자·터치 영향: 같은 모바일 회귀가 재발해도 CI가 통과할 수 있다.
- 난이도·위험: 시각 회귀 자동화 M, 수동 브라우저 실측 S.
- 초기 의견: 이번 변경의 360×800·320×568 실화면 검증은 필수. 새 CI 체계는 범위 판단 필요.

### Planner

- 문제와 근거: PR #13은 목록 상태 탭만 해결하고 상세의 권한·터치·겹침은 남긴다.
- 사용자·터치 영향: 알림 설정 성공 여부와 조작성이 핵심 acceptance를 충족하지 못한다.
- 난이도·위험: 로직 M, 배치·CSS S.
- 초기 의견: 권한 성공 시에만 구독, 상태 동기화, 44px 스위치, 내비와 0px 겹침을 합격 기준으로 제안.

## Claude Code 독립 검토

- 문제 우선순위:
  1. 권한 결과 확인 없이 구독되는 거짓 성공 상태.
  2. 권한 CTA와 고정 내비게이션의 6px 겹침.
  3. 34px 스위치와 접근 가능한 이름 부재.
- 채택: 권한이 `granted`일 때만 구독, 알림 카드 상향, 스위치 44px 이상과 이름 추가.
- 반박: 별도 고정 CTA는 작은 화면의 고정 요소를 이중화하므로 제외.
- 보완: 외부 링크의 안전상 중요성은 유지하되 알림 행동보다 전부 앞세우지 않는다.
- 성능: 현재 수치가 양호하므로 근거 없는 최적화 제외.
- PR 의견: PR #13 분리를 제안했으나, 확인 시점에 PR #13은 이미 병합되어 새 작업 브랜치가 필요해짐.

- Inspector: 안내문 간격만 줄이면 제목 길이·외부 링크 수에 따라 겹침이 재발하므로 알림 카드 자체를 구조적으로 위로 이동해야 한다.

## 최종 리드 판단

### 채택

1. 목록에서는 권한이 `default`일 때 선행 허용 배너를 숨겨 첫 공고를 위로 올린다. `denied`·`unsupported` 경고는 유지한다.
2. 상세는 `카운트다운 → 모집공고 원문 → 알림 카드 → 청약홈·공식 홈페이지` 순서로 바꾼다.
3. 권한 요청 결과가 `granted`일 때만 구독을 저장하고, 배너와 스위치가 같은 권한 상태를 사용한다.
4. 스위치 실제 터치 영역을 최소 44px로 만들고 공고명을 포함한 접근 가능한 이름을 제공한다.

### 보류

- 자동 시각 회귀·PWA CI: 이번 변경보다 인프라 범위가 커 수동 다중 뷰포트 검증으로 제한.
- 로딩·빈 상태 `aria-live`: 현재 핵심 행동 실패보다 영향이 낮아 후속 접근성 작업으로 분리.
- PR #9 서비스워커 precache: 최신 main 재작성과 콜드 오프라인 검증이 필요.

### 제외

- 별도 고정 CTA: 기존 고정 하단 내비와 작은 화면을 이중 점유한다.
- 전역 내비·디자인·대규모 CSS 리팩터링.
- 현재 지표에 근거가 없는 성능 최적화.

### 합격 기준

- 권한이 `granted`가 아니면 구독·스냅샷과 스위치가 켜지지 않는다.
- 권한 요청 직후 배너·스위치·구독 상태가 일치한다.
- 스위치 히트박스 높이가 44px 이상이고 접근 가능한 이름이 존재한다.
- 320×568과 360×800에서 첫 공고와 상세 알림 행동이 이전보다 위에 있으며 고정 내비에 가리지 않는다.
- 타입체크·테스트·빌드·브라우저 오류·가로 넘침 검증을 통과한다.

## 구현 및 검증 결과

- 목록의 기본 권한 요청 배너를 숨기고 `denied`·`unsupported` 경고는 유지했다.
- 상세 순서를 `카운트다운 → 모집공고 원문 → 알림 카드 → 보조 외부 링크`로 바꿨다.
- 권한이 `granted`일 때만 구독하며, 권한 배너와 스위치가 같은 상태를 사용한다.
- 스위치 히트박스를 `58×44`로 늘리고 공고명을 포함한 접근 가능한 이름을 추가했다.
